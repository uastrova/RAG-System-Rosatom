# Rosatom RAG
 
End-to-end RAG-система для вопросно-ответного поиска по нормативно-технической документации.
 
Проект решает задачу поиска точных ответов в корпусе PDF/DOCX-документов с возвратом источников: файла, страницы и `chunk_id`. В отличие от простого семантического поиска, система объединяет векторный поиск, BM25, объединение результатов по рангам и переранжирование с учётом нормативных сущностей: СП, НП, СТО, ПУЭ, пунктов, глав и таблиц.
 
## Что реализовано

- Извлечение текста из PDF/DOCX-документов.
- Сохранение табличных фрагментов в текстовом виде.
- Разбиение корпуса на чанки.
- Векторный поиск через FAISS и локальную модель эмбеддингов SentenceTransformer.
- Лексический поиск через BM25.
- Гибридный поиск: объединение результатов FAISS и BM25 по рангам / RRF.
- Предварительная NER-разметка чанков.
- Переранжирование с учётом сущностей: сопоставление нормативных ссылок из вопроса и чанков.
- Генерация ответа через локальную LLM, доступную как HTTP-сервер с OpenAI-совместимым API через llama.cpp.
- Консольный интерфейс для интерактивного запуска RAG и оценки качества на заранее размеченных вопросах.
- Сравнение пайплайнов поиска по Hit@K, Recall@K, Precision@K, MRR@K и nDCG@K.
 
 ## Основной pipeline
 
 ```text
  PDF/DOCX
    ↓
  Извлечение текста
    ↓
  Разбиение на чанки
    ↓
  ntd_chunks.jsonl
    ↓
  FAISS-индекс + BM25-индекс
    ↓
  Вопрос пользователя
    ↓
  Кандидаты из FAISS + кандидаты из BM25
    ↓
  Объединение результатов по рангам / RRF
    ↓
  NER/entity-переранжирование
    ↓
  Top-K контекст
    ↓
  Локальная LLM через OpenAI-совместимый API llama.cpp
    ↓
  Ответ + источники
 ```

Кратко:

```text
FAISS + BM25 + объединение по рангам + NER/entity-переранжирование + LLM
```
 
Важно: в текущей реализации NER-модель используется для предварительной разметки чанков корпуса. Сущности на стороне пользовательского вопроса извлекаются регулярными выражениями на основе правил, потому что нормативные ссылки вроде `СП 484.1311500.2020`, `НП-089-15`, `ПУЭ`, `п. 6.6.12`, `таблица 3` хорошо формализуются и устойчиво извлекаются без запуска модели на коротком вопросе.

 ## Что не хранится в git
 
 В репозитории не хранятся:

 - исходные PDF/DOCX-документы;
 - извлечённые тексты;
 - чанки;
 - FAISS-индекс;
 - предсказания NER;
 - модель эмбеддингов;
 - NER-модель;
 - GGUF-модель LLM;
 - реальные рабочие NER-файлы и полный eval-набор.

В git оставлены `.gitkeep` и example-файлы, чтобы показать ожидаемую структуру директорий и форматы данных.


## Артефакты пайплайна

| Артефакт | Путь | Как создать | Для чего нужен |
|---|---|---|---|
| Исходные документы | `data/raw/ntd/` | положить PDF/DOCX вручную | входной корпус НТД |
| Извлечённый текст | `data/processed/extracted_text/` | `python -m rosatom_rag.ingest.extract_pdf_text` и `python -m rosatom_rag.ingest.extract_docx_text` | вход для чанкинга |
| Чанки корпуса | `data/processed/chunks/ntd_chunks.jsonl` | `python -m rosatom_rag.ingest.make_chunks` | основной корпус для FAISS, BM25 и NER |
| Embedding-модель | `models/emb/sbert_large_nlu_ru/` | положить локально | построение FAISS и векторный поиск |
| FAISS-индекс | `data/processed/vectorstore/faiss_ntd/` | `python -m rosatom_rag.retrieval.build_faiss` | векторный поиск |
| NER-модель | `models/ner/rubert_ner_manual_v3_full_finetune/` | обучить или положить готовую модель | извлечение нормативных сущностей из чанков |
| NER-предсказания по чанкам | `data/ner/predictions/chunk_entities_full_v2.jsonl` | `python -m rosatom_rag.ner.predict_ner --output data/ner/predictions/chunk_entities_full_v2.jsonl` | промежуточный файл для метаданных сущностей |
| метаданные NER/entity | `data/ner/entity_index/chunk_ner_metadata_v2.jsonl` | `python -m rosatom_rag.retrieval.build_chunk_ner_metadata` | переранжирование с учётом сущностей |
| Вопросы для оценки | `data/eval/questions.jsonl` | подготовить вручную | оценка качества |
| Разметка релевантности для оценки | `data/eval/qrels.jsonl` | подготовить вручную | расчёт метрик поиска |
| Результаты оценки | `data/eval/final_hybrid_ner_cli/` | `python -m rosatom_rag.cli eval ...` | запуски поиска, JSON/CSV-метрики |

 ## Настройка LLM
Проект ожидает HTTP-сервер с OpenAI-совместимым API:

```text
http://127.0.0.1:8080/v1
```

Этот сервер можно поднять через `llama.cpp` / `llama-server`.

Инструкция:
```text
docs/install_llama_cpp.md
```
 
Базовый пример запуска сервера:

```bash
./build/bin/llama-server \
  -m /models/my_model.gguf \
  --host 0.0.0.0 \
  --port 8080 \
  -ngl 999 \
  -c 8192
```


 ## Подготовка корпуса
 
 ### 1. Положить документы
 
Положите PDF/DOCX-документы в директорию:
```text
data/raw/ntd/
```
 
 ### 2. Извлечь текст
 
 ```bash
 python -m rosatom_rag.ingest.extract_pdf_text
 python -m rosatom_rag.ingest.extract_docx_text
 ```
 
Результат сохраняется в:

```text
data/processed/extracted_text/
```

 ### 3. Разбить корпус на чанки
 
 ```bash
 python -m rosatom_rag.ingest.make_chunks
 ```
 
 Результат:
 
 ```text
 data/processed/chunks/ntd_chunks.jsonl
 ```
 
 Этот файл является основным корпусом для дальнейшего поиска: по нему строится FAISS-индекс, а BM25-индекс создаётся в памяти при запуске поиска.

### 4. Проверить наличие модели эмбеддингов

Перед построением FAISS-индекса модель эмбеддингов должна лежать в директории:

```text
models/emb/sbert_large_nlu_ru/
```

### 5. Построить FAISS-индекс
 
 ```bash
 python -m rosatom_rag.retrieval.build_faiss
 ```
 
 Результат:
 
 ```text
 data/processed/vectorstore/faiss_ntd/
 ```

После этого готовы основные артефакты поиска:
 
 ```text
 data/processed/chunks/ntd_chunks.jsonl
 data/processed/vectorstore/faiss_ntd/
 ```
 
BM25 не требует отдельной команды построения: индекс создаётся автоматически из `ntd_chunks.jsonl` при первом BM25-поиске.

 ## Подготовка NER-сущностей для поиска
 
 ### 1. Проверить наличие NER-модели
 
 NER-модель должна лежать в директории:
 
 ```text
 models/ner/rubert_ner_manual_v3_full_finetune/
 ```
 
 Модель не хранится в git. Её нужно либо обучить отдельно, либо положить готовую модель в эту директорию.
 
 ### 2. Получить NER-предсказания для чанков

 ```bash
 python -m rosatom_rag.ner.predict_ner \
   --output data/ner/predictions/chunk_entities_full_v2.jsonl
 ```
 
Результат:

```text
data/ner/predictions/chunk_entities_full_v2.jsonl
```

### 3. Собрать NER/entity metadata для reranking

 ```bash
 python -m rosatom_rag.retrieval.build_chunk_ner_metadata
 ```
 
 Результат:

 ```text
 data/ner/entity_index/chunk_ner_metadata_v2.jsonl
 ```
 
 Именно этот файл используется при `hybrid_ner` retrieval.

 ## Запуск RAG
 
 Перед запуском должны быть готовы:
 
 ```text
 data/processed/chunks/ntd_chunks.jsonl
 data/processed/vectorstore/faiss_ntd/
 data/ner/entity_index/chunk_ner_metadata_v2.jsonl
 ```
 
 ### Консольное меню
 
 ```bash
 python -m rosatom_rag.cli
 ```
 
 Меню:

```text
================================================================================
ROSATOM RAG MENU
================================================================================
1 - Получить ответ от RAG системы на свой вопрос
2 - Запустить сравнение пайплайнов без NER и с NER
0 - Выход
```
 
 ### Задать вопрос без меню
 
 ```bash
 python -m rosatom_rag.cli ask \
   --question "Согласно СП 484.1311500.2020, какой радиус контроля теплового точечного пожарного извещателя при высоте перекрытия до 3,5 м?"
 ```

## Исследование вклада NER в пайплайн

Сравниваются два пайплайна поиска:

```text
базовый:      FAISS + BM25 + объединение результатов по рангам
улучшенный:   FAISS + BM25 + объединение результатов по рангам + NER/entity-переранжирование
```

 
 ### Формат вопросов
 
 ```jsonl
 {"id": 1, "discipline": "Электротехника", "question": "Согласно СП 484.1311500.2020, какой радиус контроля теплового точечного пожарного извещателя при высоте перекрытия до 3,5 м?", "source_doc_hint": "СП 484.1311500.2020"}
 ```
 
 ### Формат релевантных чанков для вопросов
 
 ```jsonl
 {"id": 1, "relevant_chunk_ids": ["chunk_id_from_ntd_chunks_jsonl"]}
 ```
 
 Примеры файлов:
 
 ```text
 data/eval/questions.example.jsonl
 data/eval/qrels.example.jsonl
 ```
 
 ### Запустить сравнение пайплайнов
 
 ```bash
 python -m rosatom_rag.cli eval \
   --questions data/eval/questions.jsonl \
   --qrels data/eval/qrels.jsonl \
   --output-dir data/eval/final_hybrid_ner_cli
 ```
 
Консольный интерфейс запускает модуль оценки со следующими основными параметрами:

```text
K        : 10
K_VALUES : [1, 3, 5, 10]
FAISS_K  : 200
BM25_K   : 200
CANDIDATES_K: 200
RRF_K    : 60
NER_WEIGHT: 0.05
```

Результаты сохраняются в:

```text
data/eval/final_hybrid_ner_cli/runs/
data/eval/final_hybrid_ner_cli/metrics/
```

## Ограничения

- Корпус НТД, модели и индексы не публикуются в git из-за размера и ограничений на данные.
- Качество ответа зависит от качества извлечения текста из PDF/DOCX.
- BM25-индекс создаётся в памяти при запуске поиска.
- LLM не обучается в рамках проекта: используется локальный запуск модели через OpenAI-совместимый API.
- Переранжирование с учётом сущностей усиливает поиск по формальным нормативным ссылкам, но не заменяет полноценную семантическую оценку релевантности.
